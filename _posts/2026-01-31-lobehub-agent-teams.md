나는 LobeHub의 방향을 꽤 좋아한다. **‘에이전트를 일의 단위로 다룬다’**는 메시지는 지금 AI 제품들이 놓치고 있는 핵심을 정확히 찌른다. 대부분의 챗봇은 작업 단위가 “대화”에 머물러 있다. LobeHub는 그 틀을 깨고 “팀”으로 확장한다.

다만 이 야심이 항상 좋은 제품 경험으로 이어지지는 않는다. 범위가 너무 넓고, 실제 유저의 *핵심 작업 흐름*이 어디에 있는지 흐려질 위험이 있다. 이 글은 README와 공개 문서 기준으로 LobeHub를 길게 정리하고, 내가 본 장점과 불안 요소를 솔직히 이야기하는 리뷰다.

## 배경: 왜 ‘에이전트 팀’이 중요한가

오늘의 AI 제품은 대부분 단일 에이전트 중심이다. 컨텍스트는 세션에 갇혀 있고, 협업은 수동 복붙이다. 이건 업무 속도를 끌어올리는 데 치명적이다. 이 문제를 해결하려면 에이전트가 “한 번 쓰고 버리는 도구”가 아니라 **지속적으로 협업하는 팀원**이 되어야 한다.

LobeHub는 이 점을 정면으로 다룬다. README에서 말하는 핵심은 명확하다. *“Agents as the unit of work.”* 이 문장은 기능 목록보다 중요하다. 방향을 설명해 주기 때문이다.

## 문제 정의: 기존 챗 UI가 가진 구조적 한계

LobeHub가 지적하는 문제는 단순하다. 지금의 챗봇은 “컨텍스트가 얕고”, “대화가 흩어지고”, “핸드오프가 수동”이다. 이 세 가지는 모두 생산성을 깎는다. 다시 말해 **AI가 일을 도와주는 게 아니라 업무 흐름을 분리하는 도구**가 되는 순간이 많다.

이 문제는 개인 사용자보다 팀에서 더 커진다. 대화가 늘어날수록 “누가 무엇을 맡고 있는지”가 불명확해진다. 이 지점이 LobeHub가 노리는 진짜 시장이다.

## LobeHub가 내세우는 해법 (요약)

README 기준으로 보면, LobeHub의 해법은 크게 세 갈래다.

1) **Create**: Agent Builder로 “팀”을 쉽게 만든다.
2) **Collaborate**: Agent Groups로 협업 단위를 확장한다.
3) **Evolve**: Personal Memory로 관계를 장기화한다.

### 내가 특히 좋게 본 부분

- **Agent Builder + Agent Groups 조합**은 분명한 게임 체인저다. 에이전트를 “한 개”가 아니라 “팀”으로 설계하는 순간 생산성 곡선이 바뀐다.
- **화이트박스 메모리**는 강력한 메시지다. 기억을 “숨기지 않고 편집 가능하게 만든다”는 건 신뢰를 만드는 구조다.
- **Self-hosting 루트**가 잘 정리돼 있다. 개인이나 팀이 빠르게 붙어서 실험할 수 있다.

That said, 이걸 *실제 일*로 연결하는 문서는 조금 더 다듬어질 필요가 있다. 기능은 좋아 보이지만, “왜 이게 내 하루를 바꿔주는지”를 아직 충분히 설득하지는 못한다.

## 기술 관점에서 본 구성과 확장성

LobeHub는 한 프로젝트가 아니라 **에코시스템**이다. UI 컴포넌트, 아이콘, TTS, 린트 도구 등으로 생태계를 넓히고 있다. 이건 “툴킷이 있는 플랫폼”이 될 가능성을 만든다.

예를 들어, 다음처럼 UI/아이콘 패키지들이 별도로 존재한다.

```bash
# UI 컴포넌트 패키지
npm i @lobehub/ui

# 아이콘 패키지
npm i @lobehub/icons
```

이런 구조는 개발자에게 꽤 매력적이다. LobeHub를 단순 제품이 아니라 **레퍼런스 스택**으로 만들 수 있다.

## 직접 써볼 때 필요한 실제 진입 포인트

README에 있는 Self-hosting 가이드는 꽤 실용적이다. 특히 Docker 가이드는 “바로 돌려볼 수 있게” 되어 있다.

```bash
# 1) 스토리지 폴더 생성
mkdir lobe-chat-db && cd lobe-chat-db
```

```bash
# 2) 기본 인프라 초기화
bash <(curl -fsSL https://lobe.li/setup.sh)
```

```bash
# 3) 서비스 실행
docker compose up -d
```

또한 환경 변수 설정이 명확하게 정리되어 있다.

```bash
# 필수
export OPENAI_API_KEY="sk-xxxxxx...xxxxxx"

# 선택
export ACCESS_CODE="your-long-password"
export OPENAI_PROXY_URL="https://api.openai.com/v1"
```

개인적으로는 이 “바로 설치되는 경험”이 가장 중요한 데, LobeHub는 이 부분에서 꽤 잘해 놓은 편이다.

## 내가 보는 리스크: 범위가 너무 넓다

여기서부터는 나의 의견이다. **LobeHub는 너무 큰 꿈을 말하고 있다.** “일과 삶을 위한 최고의 공간”이라는 슬로건은 멋지지만, 동시에 제품 범위를 과도하게 넓힌다. 이렇게 넓은 범위는 두 가지 위험을 만든다.

1) **핵심 사용 시나리오가 흐려진다.** 사람들은 “어떤 문제를 제일 잘 해결하는지”를 알고 싶어 한다.
2) **경쟁사와의 비교 기준이 애매해진다.** Notion, Slack, ChatGPT, 리서치툴 등과 전부 겹친다.

또 하나, **라이선스는 반드시 확인해야 한다.** LobeHub는 표준 MIT/Apache가 아니라 *LobeHub Community License*를 사용한다. 개인 실험에는 충분하지만, 상업적/배포 목적이라면 조건을 꼼꼼히 읽어야 한다.

이 문제를 해결하려면 **가장 강력한 워크플로우 하나를 끝까지 파고드는 전략**이 필요하다. 예를 들어 “콘텐츠 팀”이나 “PM팀”처럼 구체적인 팀을 기준으로 한 “최적화된 템플릿 경험”이 있으면 좋겠다.

## 그럼에도 불구하고, 나는 이 방향이 맞다고 본다

AI가 직업을 대체하는지, 보조하는지에 대한 논쟁은 끝이 없다. 하지만 실전에서 중요한 건 **“사람이 AI를 어떻게 팀으로 운영하느냐”**다. 이 지점에서 LobeHub는 다른 제품들보다 훨씬 정직하고 야심차다.

게다가 LobeHub는 *화이트박스 메모리*와 같은 개념을 내세운다. 이건 단순히 기술 문제가 아니라 철학 문제다. **기억은 숨겨진 블랙박스가 아니라 편집 가능한 구조여야 한다.** 이 철학을 유지한다면 LobeHub는 신뢰 기반의 AI 협업 툴로 성장할 가능성이 높다.

## 내가 직접 실험해 보고 싶은 것들

이건 개인 계획에 가깝지만, LobeHub가 가진 가능성을 확인하려면 아래 테스트를 해봐야 한다.

- **하나의 팀을 위해 에이전트를 3~5개 구성**하고, 역할이 실제로 분리되는지 확인하기
- **스케줄 기능**을 사용해 “정해진 시간에 작업이 수행되는지” 검증하기
- **화이트박스 메모리** 편집이 실제 워크플로우에 얼마나 영향을 주는지 측정하기

이게 잘 된다면, LobeHub는 단순한 챗 UI가 아니라 **AI 팀 운영 플랫폼**으로 확실히 자리를 잡을 수 있다.

## 결론: LobeHub는 ‘큰 실험’이고, 나는 이 실험에 베팅한다

LobeHub는 과감하다. 그리고 과감함은 언제나 리스크다. 하지만 나는 이 리스크가 가치 있는 종류라고 본다. 지금 시장에서 필요한 건 “더 나은 챗봇 UI”가 아니라 **AI 팀을 운영하는 구조**다. LobeHub는 그 구조를 만들고 있다.

정리하면 이렇다.

- **장점**: 팀 단위 설계, 에코시스템 확장, 빠른 셀프호스팅
- **리스크**: 너무 큰 범위, 핵심 시나리오 부재 가능성
- **내 결론**: 방향성은 맞고, 현실적인 워크플로우 설계만 강화되면 큰 파급력이 있다

나는 앞으로도 이런 “에이전트 팀” 지향 제품을 계속 지켜볼 생각이다. 그리고 LobeHub는 그 중심에 있다. 지금 바로 써보든, 조금 더 기다리든, 최소한 이 실험이 어디까지 가는지는 주기적으로 확인할 가치가 있다.

## 체크리스트

- 내가 해결하려는 일의 단위가 “대화”인지 “팀”인지 다시 점검했나?
- 에이전트에게 맡길 수 있는 작업과 인간이 주도해야 할 작업을 분리했나?
- 메모리와 기록을 “편집 가능한 구조”로 설계했나?
- 도구를 늘리는 데 집중하기보다 워크플로우를 먼저 만들었나?
