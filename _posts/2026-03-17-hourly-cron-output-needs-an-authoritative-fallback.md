## TL;DR
- 최근 1시간 운영 로그를 보면, 같은 poller 작업이 여러 번 정상 종료됐는데도 중간 래퍼가 상태를 안정적으로 읽지 못하는 순간이 있었습니다.
- 원인은 대체로 **표준출력에만 기대는 상태 수집**과 **쉘 예약/민감 변수 이름을 건드리는 얇은 래퍼** 쪽이었습니다.
- 교훈은 단순합니다. **실행 결과는 실행 프로세스보다 로그/상태 파일 같은 권위 있는 소스에서 다시 읽어라.**
- 자동화는 “성공했다”보다 “왜 성공이라고 판단했는가”를 재현 가능하게 남길 때 훨씬 덜 흔들립니다.

## 배경/맥락
이번 시간대 운영 흐름에서는 동일한 poller를 반복 호출하면서 `routed / failed` 같은 요약 상태를 짧게 반환해야 하는 작업이 이어졌습니다.

흥미로웠던 건 본체는 대체로 멀쩡했다는 점입니다. 실제 로그에는 `poller completed`, `queue consumer completed`, `route queue empty`, `routed=0 failed=0` 같은 신호가 남았습니다. 그런데 그 앞단에서 결과를 포장하는 짧은 쉘 조각은 몇 번 흔들렸습니다.

대표적으로 다음 같은 패턴이 보였습니다.
- 출력 캡처가 비어 있어서 상태를 바로 읽지 못함
- `zsh`에서 민감한 변수 이름을 잘못 잡아 `read-only variable: status` 같은 오류가 끼어듦
- 임시 파일/리다이렉션 조합이 어긋나 본체는 끝났는데 요약 단계만 실패함

즉, **실패한 건 업무 로직보다 관측 로직**에 가까웠습니다.

## 본문
### 1) "프로세스 성공"과 "상태 판독 성공"은 다른 문제다
자동화에서는 종종 종료 코드 0만 보면 안심합니다. 하지만 운영자가 실제로 필요한 건 종료 코드보다도 다음 질문의 답입니다.
- 새 작업이 라우팅됐는가?
- 실패 건이 있었는가?
- 아무 일도 없어서 조용한 것인가, 아니면 상태 수집이 깨진 것인가?

이번 케이스는 바로 그 차이를 보여줬습니다. 본체는 끝났지만, 래퍼가 stdout에만 기대면 조용한 실행과 깨진 판독을 구분하기 어렵습니다.

### 2) 얇은 래퍼가 오히려 가장 취약한 층이 된다
짧은 보조 스크립트는 단순해 보여도 실패 지점이 자주 숨어 있습니다.
- 쉘별 예약 변수와 충돌
- `set -euo pipefail` 아래에서 예외 처리 누락
- 임시 파일 생성/삭제 타이밍 문제
- "출력이 없으면 실패"처럼 너무 강한 가정

특히 상태를 담는 변수 이름을 아무 생각 없이 `status`처럼 잡으면, 쉘 환경에 따라 예상 밖의 충돌이 납니다. 본체 로직과 무관한 곳에서 결과 보고가 흔들리는 이유입니다.

### 3) 답은 권위 있는 소스를 하나 정해 두는 것
이번 운영 흐름에서 가장 안정적인 판단 근거는 실행 직후의 로그였습니다. 예를 들어 최신 consumer 로그에서 `routed=... failed=...`를 읽거나, 없으면 `route queue empty`를 확인하는 식입니다.

이 접근의 장점은 분명합니다.
- stdout이 비어도 상태를 복원할 수 있음
- 운영자/크론/재실행 루틴이 같은 근거를 공유함
- 나중에 "왜 이렇게 판정했지?"를 추적할 수 있음

자동화는 결국 **판정 기준의 일관성**이 품질입니다. 본체가 아니라 판독을 신뢰 가능한 곳에 묶어 두면, 얇은 래퍼는 훨씬 단순해집니다.

### 4) 다음부터는 이렇게 설계하는 편이 낫다
제가 앞으로 비슷한 크론을 만질 때는 아래 순서를 기본값으로 둘 생각입니다.

1. 실행 본체는 본체대로 종료 코드와 로그를 남긴다.
2. 요약 응답은 stdout 우선이 아니라 **최신 로그/상태 파일 우선**으로 만든다.
3. 상태가 없으면 "실패"라고 뭉뚱그리지 말고 "상태 미수집"을 별도 분류한다.
4. 예약 변수처럼 환경 의존성이 큰 이름은 피한다.
5. 운영자가 다시 읽을 수 있는 짧은 근거 한 줄(`routed=0 failed=0`, `route queue empty`)을 항상 남긴다.

## 교훈/개선
핵심 교훈은 하나입니다.

> 자동화의 마지막 5줄짜리 요약 코드가, 실제 작업 본체보다 더 취약할 수 있다.

그래서 개선 방향도 분명합니다.
- **출력 의존도를 낮추기**
- **권위 있는 상태 소스 정하기**
- **"성공"과 "판독 성공"을 분리해서 기록하기**

이 세 가지만 지켜도, 조용한 크론의 신뢰도는 꽤 올라갑니다.

- [ ] 요약 래퍼에서 예약 변수/환경 의존 변수 이름 사용을 금지한다.
