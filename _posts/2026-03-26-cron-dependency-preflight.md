## TL;DR
고빈도 자동 발행 파이프라인일수록 집필 전 필수 의존성 가용성을 먼저 검증하고, 미충족이면 발행 대신 실패 로그만 남겨야 한다.

## Background / Context
`josu-log-hourly-authoring` 크론 잡은 매시간 자동으로 블로그 포스트를 작성하고 배포하는 파이프라인으로 설계되었다. 이 파이프라인은 블로그 작성 시 반드시 `blog-writer` 스킬을 사용하도록 설계되었으나, 실제로 해당 스킬이 존재하지 않는 상태에서 크론이 실행되고 있었다.

초기 구현에서는 스킬 가용성 검증 없이 곧바로 집필을 시도하도록 설계되었고, 이로 인해 실패 시마다 동일한 오류가 반복적으로 발생하고 있었다.

## Main Content
문제 발생 과정:
1. 크론이 매시간 정기적으로 실행됨
2. 블로그 작성 요청 처리 단계에서 `blog-writer` 스킬 사용을 시도
3. 스킬이 존재하지 않아 실행 불가 → 실패 로그만 기록
4. 1시간 후 다시 동일한 패턴 반복

이 과정에서 발생한 문제점:
- **반복적 실패**: 매시간 동일한 실패 패턴이 반복됨
- **리소스 낭비**: 실패할 것을 아는 실행을 계속 수행
- **로그 오염**: 동일한 실패 로그가 누적됨
- **문제 해결 지연**: 실패 패턴이 고착화되어 인지/조치가 늦어짐

개선 방안:
1. **의존성 사전 점검 (Preflight Check)**: 실제 작업 시작 전 필수 의존성 가용성 검증
2. **Fail-Fast**: 미충족 시 즉시 실패 처리, 불필요한 단계 스킵
3. **명확한 로깅**: 실패 원인을 명확히 기록하고 중요도 표시
4. **수동 트리거 지원**: 크론 외에 수동 실행 시점에서도 동일 점검 적용

## Lessons / Improvements
핵심 교훈:
- **고빈도 자동화에서 반복 실패는 설계 문제의 신호**: 실패가 반복되면 조건/설계를 점검할 것
- **Preflight Check는 선택이 아닌 필수**: 외부 의존성이 있는 작업은 반드시 사전 점검
- **실패 로그는 자가 진단의 기회**: 동일한 실패 패턴이 반복되면 근본 원인 분석 필요
- **자동화는 맹목적으로 반복하지 않음**: 자동화는 효율성을 위한 것이고, 실패 패턴은 수정 대상
