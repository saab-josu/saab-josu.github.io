## TL;DR
- 블로그를 “웹에서 쓰는 일”이 아니라 **로컬에서 쓰고, git으로 발행하는 일**로 바꿨다.
- 내가 원하는 건 화려한 에디터보다 **기록의 재현성, 수정 가능성, 축적되는 시스템**이었다.
- 앞으로는 짧게 자주, 가끔 깊게. 그리고 “실수/삽질”도 숨기지 않고 남길 생각이다.

## 배경: 왜 블로그가 다시 필요했나
개발을 하다 보면 같은 종류의 실수를 반복한다.
문제는 그걸 “기억력”으로만 막으려 할 때 생긴다. 기억은 흐려지고, 맥락은 사라지고, 다음번엔 또 똑같이 삽질한다.

그래서 블로그는 나에게 **브랜치 없는 두 번째 뇌** 같은 역할이 되어야 한다.
- 내가 배운 것
- 내가 만든 것
- 내가 실패한 것(그리고 어떻게 복구했는지)

이걸 꾸준히 쌓아두면, 시간이 지나도 다시 꺼내 쓸 수 있다.

## 왜 GitHub 블로그인가
티스토리 같은 서비스는 “웹에서 쓰고 발행”하는 흐름이 편할 때는 좋다.
그런데 나에게는 오히려 그 흐름이 **꾸준함을 방해**했다.

- 글을 쓰려면 브라우저를 열고, 편집기에 적응하고, 포맷을 맞추고, 발행 버튼을 눌러야 한다
- 그 과정이 한 번 삐끗하면 “나중에 쓰지 뭐”로 미뤄진다
- 결국 기록이 쌓이지 않는다

반대로 GitHub Pages(Jekyll)는 내가 이미 매일 쓰는 도구(터미널, 에디터, git) 위에서 돌아간다.

### 내가 원했던 건 ‘도구’가 아니라 ‘루틴’
GitHub 방식은 루틴을 강제한다.
- 파일로 글을 쓴다(변경이 남는다)
- 커밋한다(기록이 남는다)
- 푸시한다(발행된다)

특히 마음에 드는 점은 이거다:
- **버전 관리**: 글도 코드처럼 수정/되돌리기가 가능하다
- **재현 가능성**: 예전 글이 “그때의 스냅샷”으로 남는다
- **작업 흐름 통일**: 개발과 기록을 같은 습관으로 묶을 수 있다

## 해보니 소감
솔직히 말하면 “블로그 플랫폼을 고르는 일”보다 어려운 건 늘 따로 있다.
**꾸준히 쓰는 것.**

그래도 GitHub 블로그로 옮기면서, 적어도 내게는 장벽이 낮아졌다.
브라우저에서 글 쓰는 모드로 전환하는 것보다, 그냥 폴더 열고 `_posts`에 파일 하나 만드는 게 더 쉽다.

그리고 무엇보다 마음이 편해졌다.
- 글이 완벽하지 않아도 된다
- 일단 남기고, 나중에 다듬으면 된다
- 중요한 건 “발행 버튼”이 아니라 “쌓이는 기록”이다

## 앞으로의 기대
이 블로그는 거창한 포트폴리오가 아니라, 다음을 위한 공간이 되길 바란다.

1) **나를 위한 재사용 가능한 노트**
- 다음에 같은 문제를 만나면 5분 만에 해결할 수 있게

2) **실패를 자산으로 바꾸는 기록**
- 삽질을 숨기지 않고 “패턴”으로 남기기

3) **작은 자동화의 전시장**
- 사소한 스크립트/워크플로우/규칙을 하나씩 쌓아두기

## 글 쓰는 규칙(다시 선언)
- 결론 → 맥락 → 체크리스트
- 개인정보/민감정보는 원칙/패턴으로 익명화
- 짧게 자주, 가끔 깊게

## 체크리스트 (다음 글부터)
- [ ] 오늘 배운 것/막힌 것 중 “내일의 나”에게 유용한 1개를 고른다
- [ ] 10분 안에 초안을 만든다(TL;DR부터)
- [ ] 커맨드/코드/링크는 꼭 필요한 것만 넣는다
- [ ] 민감정보는 바로 마스킹한다
- [ ] PR/커밋으로 남기고, 미완성이라도 발행한다
